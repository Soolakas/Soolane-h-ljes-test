# Geometry Battles - Menu System Implementation Plan

## Overview
Implement a state-based menu system for the Pygame game "Geometry Battles" with mouse-only navigation, full settings, and an extensible architecture for future features.

## Architecture Design

### State Machine Pattern
Create a game state manager that handles transitions between:
- `MENU` â†’ Main menu screen
- `SETTINGS` â†’ Settings screen  
- `UPGRADES` â†’ Upgrades tab (placeholder)
- `PLAYING` â†’ Active gameplay (current game loop)
- `PAUSED` â†’ Pause overlay during gameplay
- `GAME_OVER` â†’ Game over screen with restart option

### New File Structure
```
game.py              (modified - integrates state machine)
menu/
  __init__.py
  state_manager.py   - GameState enum + transitions
  ui/
    __init__.py
    button.py        - Reusable button component
    panel.py         - UI panel/container
    slider.py        - Settings slider component
    label.py         - Text label component
  screens/
    __init__.py
    base_screen.py   - Abstract base class for all screens
    main_menu.py     - Title + Play/Settings/Upgrades/Exit
    settings.py      - Full settings (volume, controls, display, gameplay)
    upgrades.py      - Placeholder for future upgrades
    pause.py         - Pause overlay
    game_over.py     - Game over with score + restart
config/
  __init__.py
  settings_store.py  - Persistent settings (save/load JSON)
```

## Implementation Steps

### Step 1: Create State Manager (`menu/state_manager.py`)
- Define `GameState` enum: `MENU`, `PLAYING`, `PAUSED`, `SETTINGS`, `UPGRADES`, `GAME_OVER`
- Create `StateManager` class with:
  - `current_state` property
  - `change_state(new_state)` method with cleanup/initialization hooks
  - Stack-based navigation (push/pop for sub-screens like settings)

### Step 2: Create Base UI Components (`menu/ui/`)
- **`button.py`**: `UIButton` class
  - Constructor: `text, position, size, callback, font_size, colors`
  - `handle_event(event)` - return True if clicked
  - `draw(screen)` - render with hover/active states
  - Support for hover color changes

- **`label.py`**: `UILabel` class  
  - Constructor: `text, position, font_size, color`
  - `draw(screen)` - render static text

- **`slider.py`**: `UISlider` class
  - Constructor: `position, width, value, min_val, max_val, label, callback`
  - `handle_event(event)` - drag to change value
  - `draw(screen)` - render track + handle + label
  - Return current value

- **`panel.py`**: `UIPanel` class (optional, for grouping)
  - Background rectangle with optional border
  - Contains list of child UI elements

### Step 3: Create Settings Store (`config/settings_store.py`)
- `GameSettings` class with defaults:
  - Audio: `music_volume=0.7`, `sfx_volume=0.8`
  - Display: `fullscreen=False`, `resolution=(1280, 720)`
  - Controls: key bindings dict (WASD defaults, mouse aim)
  - Gameplay: `show_fps=False`, `particle_quality="high"`
- `save()` / `load()` methods using JSON file (`settings.json`)
- Singleton pattern or module-level instance

### Step 10: Integrate into `game.py`
- **Restructure the main loop:**
  ```python
  state_manager = StateManager()
  settings = load_settings()
  
  # Initialize all screens
  screens = {
      GameState.MENU: MainMenuScreen(state_manager, settings),
      GameState.SETTINGS: SettingsScreen(state_manager, settings),
      GameState.UPGRADES: UpgradesScreen(state_manager, settings),
      GameState.PLAYING: None,  # Game logic, not a screen
      GameState.PAUSED: PauseScreen(state_manager, settings),
      GameState.GAME_OVER: GameOverScreen(state_manager, settings),
  }
  
  while running:
      events = pygame.event.get()
      for event in events:
          if event.type == pygame.QUIT:
              running = False
          elif state_manager.current_state in screens:
              screens[state_manager.current_state].handle_event(event)
      
      # Route update/draw based on state
      if state_manager.current_state == GameState.PLAYING:
          # Run existing game logic
          update_game_logic(dt)
          draw_game()
          if game_over:
              state_manager.change_state(GameState.GAME_OVER)
      elif state_manager.current_state == GameState.PAUSED:
          draw_game()  # Draw game behind pause overlay
          screens[GameState.PAUSED].handle_events(events)
          screens[GameState.PAUSED].update(dt)
          screens[GameState.PAUSED].draw(screen)
      else:
          screen = screens[state_manager.current_state]
          screen.handle_events(events)
          screen.update(dt)
          screen.draw(screen)
      
      pygame.display.flip()
      dt = clock.tick(60) / 1000
  ```

- **Escape key handling:** During `PLAYING`, Escape toggles `PAUSED` state
- **Preserve all existing game logic** - wrap it in a `GameState.PLAYING` conditional

## Key Design Decisions

1. **Component-based UI** - Reusable UIButton, UISlider, etc. make adding future screens trivial
2. **Screen abstraction** - Each screen is self-contained, easy to modify independently  
3. **Settings persistence** - JSON file for settings means preferences survive between sessions
4. **State stack** - Push/pop pattern allows returning from sub-screens (settings) without losing context
5. **Separation of concerns** - Game logic stays in game.py, UI logic in menu/ package
6. **No external assets required** - Uses pygame.font.SysFont like existing code, no new dependencies

## Future Extensibility Hooks

- **New screens**: Just create new screen class inheriting from BaseScreen
- **New settings**: Add to GameSettings class, add UI in settings.py
- **Upgrades system**: The upgrades screen structure is ready - just fill with upgrade items
- **New UI components**: Extend the ui/ package with dropdowns, checkboxes, etc.
- **Screen transitions**: Add fade/slide effects to StateManager.change_state()
- **Sound system**: Volume sliders already wired to settings, just add pygame.mixer calls
- **Controller support**: Controls section in settings ready for gamepad keybinds
